# Persistent repo memory

## Project identity

- This is a PUCV Optimización Computacional project.
- Goal: progressively approach a rigorous replication of Duran-Mateluna, Ales & Elloumi (2023), “An efficient Benders decomposition for the p-median problem”, while keeping local claims tied to evidence.

## Current verified state

- Benders/F4-style lazy-cut method derived from F3 is implemented.
- Implemented core: master variables `y_j` and `theta_i`; objective `min sum_i theta_i`; constraint `sum_j y_j = p`; Phase 1 LP relaxation; Phase 2 binary branch-and-Benders-cut; Gurobi lazy callbacks; separator using sorted sites `S_i` and `ktilde_i`.
- Locally verified experiments before/through this branch: `toy1`, OR-Library `pmed1`–`pmed15`, `rl1304` Table 2 subset, `kroA100`, warm/cold comparison, separator tests, and branch-local 300-second Benders wrapper campaign when `results/benders_300s_campaign.csv` plus `results/logs/benders_300s/*.log` are present.
- Current audited missing/not-yet-run components: Zebra, PopStar, constraint reduction, reduced-cost fixing, monolithic C benchmark, large/huge TSPLIB campaigns, BIRCH, RW large, ODM.

## Target replication roadmap

Missing components are roadmap items, not permanent prohibitions. Future experimental branches may implement or run:

- external 300s benchmark campaign;
- monolithic F1 baseline in C;
- potentially F3/F4 monolithic baseline if feasible;
- larger OR-Library and TSPLIB campaigns;
- synthetic stress tests;
- gap trace logging;
- PopStar or alternative warm-start heuristic if time allows;
- Zebra only if source/build is available and reproducible.

## Claim discipline

- Every numerical claim must be backed by `results/*.csv`, `results/logs/*.log`, or tests.
- Do not claim any roadmap item as completed unless code plus CSV/log/test evidence exists.
- Forbidden until supported by new evidence:
  - “we outperform Zebra”
  - “we replicated the full paper”
  - “we implemented PopStar”
  - “we compared against a monolithic C baseline”
  - “we ran huge TSP/BIRCH/ODM/RW large”
- Time-limit rule: if a result is described as 300s / 5 minutes, it must be run through a wrapper enforcing `timeout=300`.

## Branch policy

Experimental implementations should happen in separate branches:

- `exp/monolithic-baselines`
- `exp/large-instance-campaign`
- `exp/synthetic-stress`
- `exp/gap-logging`
- `exp/popstar-or-warmstart`
- `exp/zebra-comparison` only if Zebra source/build is available

## Immediate priority

Current task: make Benders results defensible under the professor’s 5-minute protocol using an external 300-second wrapper and CSV/log traceability. This does not replace future monolithic baseline, large-instance campaigns, gap tracing, synthetic stress tests, or Zebra/PopStar work.
